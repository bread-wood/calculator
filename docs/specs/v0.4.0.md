# Calculator CLI v0.4.x — User-Defined Functions

**Theme:** Reusable computation. The calculator becomes programmable.

## Overview

Extends the v0.3.x variable system with user-defined single-expression functions.
A user can define a named function with parameters, then call it anywhere in the
same expression sequence — including from within other user-defined functions.
Function bodies may reference their own parameters, named constants, built-in
functions, and any user-defined function defined earlier in the sequence. Targets
developers who need to define a formula once and apply it repeatedly within a
computation without repeating themselves.

## Success Criteria

- [ ] `calc 'def double(x) = x * 2; double(5)'` prints `10` to stdout and exits 0
- [ ] `calc 'def hyp(a, b) = sqrt(pow(a, 2) + pow(b, 2)); hyp(3, 4)'` prints `5` to stdout and exits 0
- [ ] `calc 'def square(x) = pow(x, 2); def cube(x) = x * square(x); cube(3)'` prints `27` to stdout and exits 0
- [ ] `calc 'def area(r) = pi * pow(r, 2); area(5)'` prints `78.53981633974483` to stdout and exits 0
- [ ] `calc 'x = 10; def half(n) = n / 2; half(x)'` prints `5` to stdout and exits 0
- [ ] `calc 'def f(x) = x + 1; def g(x) = f(f(x)); g(3)'` prints `5` to stdout and exits 0
- [ ] `calc 'f(5)'` where `f` is never defined prints `error: undefined function: f` to stderr and exits 1
- [ ] `calc 'def f(x) = x; f(1, 2)'` prints `error: wrong number of arguments: f expects 1` to stderr and exits 1
- [ ] `calc 'def f(x) = x; def f(y) = y * 2; f(3)'` prints `error: function already defined: f` to stderr and exits 1
- [ ] `calc 'def f(x) = g(x); def g(x) = x'` prints `error: undefined function: g` to stderr and exits 1 — forward references not allowed
- [ ] All v0.1.x, v0.2.x, and v0.3.x success criteria still pass
- [ ] `make test` passes clean on macOS and Linux

## Scope

**Included:**

- Function definition syntax: `def <name>(<params>) = <expression>`
- Single-expression function bodies — no multi-statement bodies inside a function
- Zero or more parameters; zero-parameter functions are valid: `def two() = 1 + 1`
- Function bodies may reference: their own parameters, named constants (`pi`, `e`),
  built-in functions, and user-defined functions defined earlier in the sequence
- Function bodies may NOT reference outer variables from the enclosing statement
  sequence — parameters and constants only
- User-defined functions callable with the same syntax as built-ins: `f(x)`
- Functions defined in order; forward references to not-yet-defined functions are
  an error
- Variables and functions share a single statement sequence; `def` statements
  and assignment statements may be freely interleaved
- All v0.3.x variables, v0.2.x functions and constants unchanged

## Failure Modes

**Function definition errors:**
- Forward reference to an undefined function in a function body → `error: undefined function: <name>` to stderr, exit 1
- Duplicate function name → `error: function already defined: <name>` to stderr, exit 1
- Shadowing a built-in function name → `error: cannot redefine built-in: <name>` to stderr, exit 1

**Function call errors:**
- Call to undefined function → `error: undefined function: <name>` to stderr, exit 1
- Wrong argument count → `error: wrong number of arguments: <name> expects <N>` to stderr, exit 1

**Variable errors (inherited):**
- Reference to undefined variable → `error: undefined variable: <name>` to stderr, exit 1
- Reassignment of named constant → `error: cannot reassign constant: <name>` to stderr, exit 1

**Arithmetic (inherited):**
- Division by zero → `error: division by zero` to stderr, exit 1
- Domain error → `error: domain error` to stderr, exit 1
- Overflow → `error: overflow` to stderr, exit 1

## Interface Contracts

**CLI:**

```
calc '<statement> [; <statement> ...]'
```

| Stream | Content |
|--------|---------|
| stdout | The result as a number (integer if whole, decimal otherwise), followed by a newline |
| stderr | `error: <description>` followed by a newline, on any failure |
| exit 0 | Expression evaluated successfully |
| exit 1 | Any error: undefined/duplicate function, wrong arguments, undefined variable, arithmetic error |

**Statement grammar (informal):**

```
program    = statement { ';' statement }
statement  = 'def' IDENT '(' [ IDENT { ',' IDENT } ] ')' '=' expression
           | IDENT '=' expression
           | expression
expression = <v0.3.x expression grammar, extended to include user-defined function calls>
```

**Scoping rules:**

| Lookup target | Visible to function body? |
|---------------|--------------------------|
| Own parameters | Yes |
| Named constants (`pi`, `e`) | Yes |
| Built-in functions | Yes |
| User-defined functions defined before this one | Yes |
| Outer variables from enclosing sequence | No |
| User-defined functions defined after this one | No |

---

## Constraints

- All v0.1.x through v0.3.x constraints remain in force
- Recursive functions are not supported in this version — a function body may not
  call itself directly or indirectly; detection of cycles is not required (forward
  reference prohibition makes direct recursion impossible by construction; mutual
  recursion via forward reference is also impossible)
- Function bodies are single expressions — no multi-statement bodies, no local
  variable assignment within a function
- The function definition and call mechanism must extend to a persistent REPL
  environment in a future version — specifically, function definitions must be
  representable as serializable values without redesigning the runtime

---

## Key Unknowns

1. **[P1]** Does the ordered, forward-reference-prohibiting function definition
   model extend to a persistent REPL where functions defined in earlier sessions
   must be visible in later ones — specifically, can the definition ordering
   constraint be relaxed or replaced without breaking the scoping model when
   persistence is introduced? The answer determines whether v0.4.x function
   semantics are the foundation for the REPL or must be revised before
   persistence is added.

---

## Non-Goals

- Recursive functions — requires cycle detection or a stack depth limit; future version
- Multi-statement function bodies with local variables
- Closures — function bodies do not capture outer variables
- Interactive REPL mode — future version
- Windows support in this version
