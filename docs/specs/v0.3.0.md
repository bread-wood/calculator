# Calculator CLI v0.3.x — Variables

**Theme:** Named values. The calculator gains memory within a single expression.

## Overview

Extends the v0.2.x function library with named variable assignment and
multi-statement expressions. A user can now name intermediate results, build
up a computation across several assignments, and reference the named values
in the final expression. Variables are scoped to a single invocation — there
is no state file and no persistence between runs. Targets developers who
outgrow single-expression calculations and need to annotate their work without
leaving the terminal.

## Success Criteria

- [ ] `calc 'x = 5; x * 2'` prints `10` to stdout and exits 0
- [ ] `calc 'x = 3; y = 4; sqrt(pow(x, 2) + pow(y, 2))'` prints `5` to stdout and exits 0
- [ ] `calc 'x = 2; y = x * x; y + 1'` prints `5` to stdout and exits 0
- [ ] `calc 'x = 5'` (last statement is an assignment) prints `5` to stdout and exits 0
- [ ] `calc 'x = pi * 2; x / pi'` prints `2` to stdout and exits 0
- [ ] `calc 'x'` where `x` is never assigned prints `error: undefined variable: x` to stderr and exits 1
- [ ] `calc 'pi = 3'` prints `error: cannot reassign constant: pi` to stderr and exits 1
- [ ] `calc 'e = 1'` prints `error: cannot reassign constant: e` to stderr and exits 1
- [ ] `calc 'x = 5; x = 6; x'` prints `6` to stdout and exits 0 — reassignment of user variables is allowed
- [ ] All v0.1.x and v0.2.x success criteria still pass
- [ ] `make test` passes clean on macOS and Linux

## Scope

**Included:**

- Named variable assignment: `x = <expression>`
- Multi-statement expressions separated by `;`
- The value of the last statement is the result; if the last statement is an
  assignment, the assigned value is the result
- Variables scoped to the single invocation — no state file, no cross-run persistence
- Named constants from v0.2.x (`pi`, `e`) treated as read-only pre-defined variables
  in the same namespace; reassignment is an error
- Variables may reference earlier variables in the same statement sequence
- All v0.2.x functions and constants unchanged

---

## Failure Modes

**Variable errors:**
- Reference to undefined variable → `error: undefined variable: <name>` to stderr, exit 1
- Reassignment of a named constant → `error: cannot reassign constant: <name>` to stderr, exit 1

**Input validation (inherited):**
- No arguments → print usage to stderr, exit 1
- More than one argument → `error: expected a single quoted expression` to stderr, exit 1
- Empty string argument → `error: empty expression` to stderr, exit 1
- Malformed expression → `error: <description>` to stderr, exit 1
- Unrecognised token → `error: unexpected token` to stderr, exit 1

**Arithmetic (inherited):**
- Division by zero → `error: division by zero` to stderr, exit 1
- Domain error → `error: domain error` to stderr, exit 1
- Overflow → `error: overflow` to stderr, exit 1

## Interface Contracts

**CLI:**

```
calc '<statement> [; <statement> ...]'
```

| Stream | Content |
|--------|---------|
| stdout | The result as a number (integer if whole, decimal otherwise), followed by a newline |
| stderr | `error: <description>` followed by a newline, on any failure |
| exit 0 | Expression evaluated successfully |
| exit 1 | Any error: undefined variable, constant reassignment, malformed input, arithmetic error |

**Statement grammar (informal):**

```
program    = statement { ';' statement }
statement  = IDENT '=' expression
           | expression
expression = <v0.2.x expression grammar, extended to include IDENT references>
```

---

## Constraints

- All v0.1.x and v0.2.x constraints remain in force
- Variables are evaluated eagerly and in order; a variable may not reference
  a name assigned later in the same statement sequence
- The variable namespace is flat — no block scoping in this version
- The scoping model must extend to function parameter scoping in the next version
  without introducing a second incompatible mechanism

---

## Key Unknowns

1. **[P1]** Does the flat, single-pass variable namespace from v0.3.x extend
   cleanly to function parameter scoping in v0.4.x — where parameters must
   shadow outer variables within the function body — or does adding functions
   require replacing the scope model rather than extending it? The answer
   determines whether v0.3.x scope design is the foundation for v0.4.x or
   a layer that must be replaced.

---

## Non-Goals

- Persistence across invocations — no state file
- Block scoping or nested scopes
- User-defined functions — v0.4.x
- Interactive REPL mode — future version
- Windows support in this version
